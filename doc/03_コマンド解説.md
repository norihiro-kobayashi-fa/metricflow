# コマンド解説：dbt run / dbt test / MetricFlow

## コマンドの実行方法

このプロジェクトは Docker コンテナ上で動作します。  
dbt コマンドはすべて以下のように `docker compose exec dbt` を前置して実行します。

```powershell
# PowerShell（Windows）
docker compose exec dbt dbt <サブコマンド>

# Bash（Mac/Linux）
docker compose exec dbt dbt <サブコマンド>
```

---

## 0. 初期セットアップの流れ

初めて使うときは以下の順序で実行します。

```powershell
# 1. Dockerコンテナを起動
docker compose up -d

# 2. dbt の外部パッケージをインストール（dbt_utils）
docker compose exec dbt dbt deps

# 3. CSVをDBに読み込む
docker compose exec dbt dbt seed

# 4. SQLモデルを実行してテーブル・ビューを作成
docker compose exec dbt dbt run

# 5. データ品質テストを実行
docker compose exec dbt dbt test
```

便利なセットアップスクリプトも用意されています：
- PowerShell: `scripts/setup.ps1`
- Bash: `scripts/setup.sh`

---

## 1. dbt deps（パッケージインストール）

```powershell
docker compose exec dbt dbt deps
```

### 何をするか

`packages.yml` に書かれた外部パッケージ（ライブラリ）をダウンロードしてインストールします。

```yaml
# packages.yml
packages:
  - package: dbt-labs/dbt_utils
    version: [">=1.0.0", "<2.0.0"]
```

このプロジェクトでは **dbt_utils v1.3.3** がインストールされます。  
インストール先は `dbt_project/dbt_packages/` フォルダです。

> `dbt deps` は最初の1回と、`packages.yml` を変更したときだけ実行すれば十分です。

## 2. dbt seed（CSVデータの読み込み）

```powershell
docker compose exec dbt dbt seed
```

### 何をするか

`seeds/` フォルダの CSVファイルを PostgreSQL にテーブルとして読み込みます。

**処理の流れ：**

```
seeds/raw_customers.csv  ─→  PostgreSQL: raw.raw_customers（テーブル）
seeds/raw_orders.csv     ─→  PostgreSQL: raw.raw_orders（テーブル）
seeds/raw_payments.csv   ─→  PostgreSQL: raw.raw_payments（テーブル）
```

`dbt_project.yml` の設定により、シードデータは `raw` スキーマに作成されます：

```yaml
seeds:
  metricflow_eval:
    +schema: raw   ← rawスキーマに作成される
```

## 3. dbt run（モデルの実行）

```powershell
docker compose exec dbt dbt run
```

### 何をするか

`models/` フォルダ内のSQLファイルを **依存関係の順番通りに実行** して、PostgreSQL にビューとテーブルを作成します。

### 実行順序と依存関係

dbt は `{{ ref('テーブル名') }}` という記法でモデル間の依存関係を自動解決します。  
実行順序は以下のようになります：

```
【第1フェーズ】ステージング（4並列で実行可能）
  stg_customers  （raw_customers に依存）
  stg_orders     （raw_orders に依存）
  stg_payments   （raw_payments に依存）
       ↓
【第2フェーズ】マート（ステージング完了後）
  customers            （stg_customers + stg_orders に依存）
  orders               （stg_orders + stg_customers + stg_payments に依存）
  metricflow_time_spine（依存なし、独立して実行）
```

### 各モデルでやっていること

#### ステージング層（`+materialized: view`）

SQLで「整形・リネーム」のみ行い、**ビュー（View）** として作成。

```sql
-- stg_orders.sql のイメージ
select
    id          as order_id,       -- 列名をわかりやすく変更
    customer_id,
    order_date,
    status      as order_status,
    amount      as order_amount,
    case
        when status = 'completed' then true
        else false
    end         as is_completed    -- フラグ列を追加
from raw_orders
```

#### マート層（`+materialized: table`）

複数テーブルを **JOIN・集計** して、実データを持つ**テーブル（Table）** として作成。

```sql
-- customers.sql のイメージ
select
    c.customer_id,
    c.full_name,
    -- 注文テーブルと結合して集計
    coalesce(o.total_orders, 0) as total_orders,
    coalesce(o.lifetime_value, 0) as lifetime_value,
    ...
from stg_customers c
left join (
    select customer_id, count(*) as total_orders, sum(order_amount) as lifetime_value
    from stg_orders
    group by customer_id
) o using (customer_id)
```

### 特定モデルだけ実行する

```powershell
# 1つのモデルだけ実行
docker compose exec dbt dbt run --select customers

# staging層だけ実行
docker compose exec dbt dbt run --select staging

# 特定モデルとその上流（依存元）をまとめて実行
docker compose exec dbt dbt run --select +customers
```

## 4. dbt test（データ品質テスト）

```powershell
docker compose exec dbt dbt test
```

### 何をするか

`schema.yml` に定義されたデータ品質チェックを実行します。  
テストに失敗すると、データに問題があることが分かります。

### 定義されているテスト一覧

| テスト対象モデル | 列名 | テスト種類 | チェック内容 |
|----------------|------|----------|------------|
| stg_customers | customer_id | unique | IDが重複していないか |
| stg_customers | customer_id | not_null | IDにNULLがないか |
| stg_customers | email | not_null | メールアドレスがNULLでないか |
| stg_orders | order_id | unique | 注文IDが重複していないか |
| stg_orders | order_id | not_null | 注文IDにNULLがないか |
| stg_orders | customer_id | not_null | 顧客IDがNULLでないか |
| stg_orders | customer_id | relationships | stg_customersのcustomer_idに存在するか（外部キー整合性） |
| stg_orders | order_status | accepted_values | ステータスが想定値のいずれかか |
| stg_payments | payment_id | not_null | 支払IDがNULLでないか |
| stg_payments | payment_status | accepted_values | ステータスが success か fail か |
| customers（マート） | customer_id | unique | 顧客IDが重複していないか |
| customers（マート） | customer_id | not_null | 顧客IDにNULLがないか |
| orders（マート） | order_id | unique | 注文IDが重複していないか |
| orders（マート） | order_id | not_null | 注文IDにNULLがないか |
| orders（マート） | customer_id | not_null | 顧客IDがNULLでないか |
| orders（マート） | order_date | not_null | 注文日がNULLでないか |
| orders（マート） | order_amount | not_null | 注文金額がNULLでないか |

### テストの種類

**unique（一意性）**
```sql
-- dbt が内部的に生成するSQL（イメージ）
select customer_id, count(*) from stg_customers
group by customer_id having count(*) > 1
-- 結果が0行 → テスト成功
```

**not_null（NULL禁止）**
```sql
select count(*) from stg_customers where customer_id is null
-- 結果が0 → テスト成功
```

**accepted_values（許容値チェック）**
```sql
select order_status from stg_orders
where order_status not in ('completed','returned','return_pending','placed','shipped')
-- 結果が0行 → テスト成功
```

**relationships（外部キー整合性）**
```sql
select customer_id from stg_orders
where customer_id not in (select customer_id from stg_customers)
-- 結果が0行 → テスト成功（全注文に対応する顧客が存在する）
```

```

Concurrency: 4 threads (target='dev')

1 of 17 START test not_null_stg_customers_customer_id ......... [RUN]
```

## 5. MetricFlow コマンド

MetricFlow は `mf` コマンドで操作します。

### 設定の検証

```powershell
docker compose exec dbt mf validate-configs
```

セマンティックモデルとメトリクスの定義に問題がないか確認します。

### メトリクス一覧の確認

```powershell
docker compose exec dbt mf list metrics
```

定義された12種類のメトリクスが一覧表示されます。

### ディメンション（集計軸）の確認

```powershell
docker compose exec dbt mf list dimensions --metrics revenue
```

`revenue`（売上合計）を集計できる軸（国別、日付別など）を一覧表示します。

### クエリの実行

```powershell
# 月別売上
docker compose exec dbt mf query --metrics revenue --group-by metric_time__month

# 国別注文件数
docker compose exec dbt mf query --metrics order_count --group-by country

# 月別・国別の売上（複数軸の組み合わせ）
docker compose exec dbt mf query --metrics revenue --group-by metric_time__month,country

# 支払方法別・完了率
docker compose exec dbt mf query --metrics completion_rate --group-by payment_method

# 累積顧客数（月次）
docker compose exec dbt mf query --metrics cumulative_customers --group-by metric_time__month
```

MetricFlow の強みは、SQLを書かずにYAMLで定義したメトリクスを **あらゆる軸で自由に集計** できる点です。

## 6. よく使うコマンドまとめ

| コマンド | 用途 | 頻度 |
|---------|------|------|
| `dbt deps` | パッケージのインストール | 初回・packages.yml変更時 |
| `dbt seed` | CSVデータの投入 | 初回・データ変更時 |
| `dbt run` | モデルの実行（ビュー・テーブル作成） | モデルSQL変更時 |
| `dbt test` | データ品質チェック | run後、定期実行 |
| `dbt build` | seed + run + test をまとめて実行 | 全体更新時 |
| `mf validate-configs` | MetricFlow定義の検証 | YAML変更時 |
| `mf list metrics` | メトリクス一覧表示 | 確認時 |
| `mf query --metrics ...` | メトリクスのクエリ実行 | 分析時 |

> **`dbt build`** は `dbt seed`（シード） + `dbt run`（実行） + `dbt test`（テスト）を一括で行う便利コマンドです。

## 7. ログとデバッグ

実行ログは `dbt_project/logs/dbt.log` に保存されます。  
エラーが発生した場合はこのファイルで詳細を確認できます。

コンパイル済みSQLは `dbt_project/target/compiled/` に出力されます。  
dbt が実際に実行したSQLを確認したい場合はここを参照してください。

# dbt Semantic Layer：概念・現状ギャップ・統合方針

## 1. dbt Semantic Layer とは

**dbt Semantic Layer（セマンティックレイヤー）** は、ビジネス指標の定義を一元化し、あらゆる下流ツール（BIツール・アプリ・API）に対して **単一の「正しい数字」** を提供するアーキテクチャ層です。

```
         ┌─────────────────────────────────────────────────────────┐
         │                   dbt Semantic Layer                    │
         │                                                         │
         │  ┌──────────────────┐    ┌───────────────────────────┐  │
         │  │ Semantic Models  │    │        Metrics            │  │
         │  │  - entities      │    │  - simple / ratio         │  │
         │  │  - dimensions    │───▶│  - cumulative / derived   │  │
         │  │  - measures      │    │                           │  │
         │  └──────────────────┘    └───────────────────────────┘  │
         │               │                      │                  │
         │               ▼                      ▼                  │
         │         MetricFlow（SQLを自動生成するクエリエンジン）       │
         └─────────────────────────────────────────────────────────┘
                  ▲                                       ▲
                  │ JDBC / GraphQL API                   │
         ┌────────┴────────┐                    ┌────────┴────────┐
         │  Tableau / Power│                    │   Custom App /  │
         │  BI / Metabase  │                    │   Python / REST │
         └─────────────────┘                    └─────────────────┘
```

### 従来の問題（セマンティックレイヤーがない場合）

```
レポートA独自の "売上" SQL ──┐
ダッシュボードB独自の "売上" SQL ─┼──▶ 数字がバラバラ
アドホッククエリCの "売上" SQL ──┘         定義の不一致
```

レポートやダッシュボードごとに指標の計算ロジックが分散し、「どの数字が正しいか」が曖昧になります。

### セマンティックレイヤーが解決すること

| 課題 | 解決策 |
|------|--------|
| 指標定義の分散・不一致 | YAMLで一元定義。全ツールが同じ定義を参照 |
| SQLの手書きコスト | MetricFlowがSQLを自動生成 |
| ディメンション組み合わせの爆発 | 任意の軸で動的に集計（定義は1つ） |
| ツール切り替えのコスト | APIが変わらないためBIツール乗り換えが容易 |

## 2. このプロジェクトの現状

### 構成済みの要素

このプロジェクトはセマンティックレイヤーの **定義層** はすでに実装されています。

| 要素 | ファイル | 状態 |
|------|---------|------|
| セマンティックモデル（orders） | `models/metrics/semantic_models.yml` | 実装済み |
| セマンティックモデル（customers） | `models/metrics/semantic_models.yml` | 実装済み |
| メトリクス定義（12種類） | `models/metrics/metrics.yml` | 実装済み |
| MetricFlowクエリエンジン | `dbt-metricflow 0.12.x` | 動作確認済み |

```
seeds/ → staging/ → marts/ → semantic_models.yml + metrics.yml
                                        ↓
                              mf query（CLIで手動実行）
```

### ギャップ：外部接続の手段がない

現状の構成では、**MetricFlowはDockerコンテナ内のCLIとして閉じており**、外部ツールからメトリクスを取得する手段がありません。

```
【現状】
BIツール → ??? → mf query（コンテナ内CLI） → PostgreSQL
              ↑
           接続手段なし

【理想】
BIツール → Semantic Layer API → MetricFlow → PostgreSQL
```

#### ギャップの詳細

| 項目 | 現状 | 理想 |
|------|------|------|
| メトリクスの取得方法 | コンテナ内で `mf query` を手動実行 | JDBC / GraphQL / REST API |
| BIツール接続 | なし（pgAdminのみ、MetricFlow定義を無視） | Tableau・Power BI等がAPI経由で直接取得 |
| プログラマティックアクセス | なし | Python / アプリからAPIクエリ |
| リアルタイム利用 | 不可 | 可能 |

## 3. 統合方針案

### 方針A：dbt Cloud Semantic Layer への移行（本番パス）

dbt Cloud（有償）に移行すると、JDBC・GraphQL APIが自動で有効になります。

```
BIツール ──JDBC/GraphQL──▶ dbt Cloud Semantic Layer API
                                      │
                              MetricFlow（クラウド管理）
                                      │
                                 PostgreSQL
```

**接続可能なツール（ネイティブ統合）：**
- Tableau / Power BI / Looker Studio
- Hex / Mode / Metabase
- Python SDK（`dbt-sl-sdk`）

**トレードオフ：**

| メリット | デメリット |
|--------|----------|
| 完全なSemantic Layer機能 | dbt Cloud Team プラン以上（有償） |
| BI各社のネイティブ統合 | クラウド依存・ローカル環境では再現不可 |
| 運用管理が不要 | 現構成（ローカルDocker）から大幅な変更が必要 |

---

### 方針B：ローカルAPIラッパーの追加（現実的な拡張）

OSS版MetricFlowは組み込みHTTPサーバーを持たないものの、**`MetricFlowClient` というPython APIを公開しています**。`mf` CLIコマンド自体がこのPython APIのラッパーとして実装されており、subprocessを経由せずに直接ライブラリとして呼び出せます。

これを利用し、現在のDocker Compose環境に **FastAPIサービス** を追加してHTTP APIとして公開する構成が取れます。

```
BIツール / Python ──HTTP──▶ FastAPI コンテナ
                                  │
                         MetricFlowClient（Python API）
                                  │
                            MetricFlow → PostgreSQL
```

**実装イメージ：**

```
docker-compose.yml
├── postgres     （既存）
├── dbt          （既存：MetricFlow実行環境）
├── pgadmin      （既存）
└── api          （追加：FastAPI）
        │
        └── GET /query?metrics=revenue&group_by=metric_time__month
                ↓
            MetricFlowClient.query(metrics=["revenue"], group_by=["metric_time__month"])
                ↓
            JSON レスポンス
```

**コードイメージ：**

```python
from metricflow_client import MetricFlowClient
from fastapi import FastAPI

app = FastAPI()
mf_client = MetricFlowClient.from_config(config_path="...")

@app.get("/metrics")
def list_metrics():
    return mf_client.list_metrics()

@app.get("/query")
def query(metrics: str, group_by: str):
    result = mf_client.query(
        metrics=[metrics],
        group_by=[group_by],
    )
    return result.to_dict()
```

**エンドポイント例：**

```
GET /metrics                           # メトリクス一覧
GET /dimensions?metrics=revenue        # 使用可能なディメンション一覧
GET /query?metrics=revenue&group_by=metric_time__month  # クエリ実行
```

**トレードオフ：**

| メリット | デメリット |
|--------|----------|
| 既存構成を壊さずに拡張 | dbt Cloud統合のような高度な最適化はない |
| ローカル完結・無償 | BIツールとの直接ネイティブ統合は不可 |
| MetricFlowの定義がそのまま機能 | |
| Python APIの直接呼び出しでオーバーヘッドなし | |
| 実装コストが低い（50〜100行程度） | |

---

## 4. 方針の比較と推奨

| 観点 | 方針A（dbt Cloud） | 方針B（APIラッパー） |
|------|:---:|:---:|
| セマンティック定義の活用 | ◎ | ○ |
| BI統合の容易さ | ◎ | △ |
| 実装コスト | 高（構成変更大） | 低 |
| 費用 | 有償 | 無償 |
| 本番利用の現実性 | ◎ | △ |

**推奨：方針B（APIラッパー）**

このデモ環境の目的（MetricFlowの評価・学習）に対して、最もバランスが良い選択肢です。

- 既存のセマンティックモデル・メトリクス定義を壊さずに拡張できる
- MetricFlowの定義が実際に外部統合に機能することを検証できる
- 将来的に方針Aへ移行する際の判断材料になる

方針Aへの移行は、このデモで価値を確認した後に検討するのが現実的な順序です。

# データ構造の解説

## 1. シードデータ（seeds/）

「シード」とは、分析の出発点となる **生の元データをCSVファイルとして用意したもの** です。  
実際のプロジェクトでは外部システムからエクスポートしたデータを想定しています。

`dbt seed` コマンドを実行すると、これらのCSVが PostgreSQL の `raw` スキーマにテーブルとして読み込まれます。

---

### raw_customers.csv（顧客データ）

**場所：** `dbt_project/seeds/raw_customers.csv`

| 列名 | 意味 |
|------|------|
| id | 顧客の一意なID |
| first_name | 名（ファーストネーム） |
| last_name | 姓（ラストネーム） |
| email | メールアドレス |
| created_at | 登録日 |
| country | 居住国（US / JP / GB / DE） |

**データ概要：** 15名のサンプル顧客（2023年1月〜5月に登録）

---

### raw_orders.csv（注文データ）

**場所：** `dbt_project/seeds/raw_orders.csv`

| 列名 | 意味 |
|------|------|
| id | 注文の一意なID |
| customer_id | どの顧客の注文か（raw_customersのidと対応） |
| order_date | 注文日 |
| status | 注文ステータス |
| amount | 注文金額（ドル） |

**ステータスの種類：**
- `placed` — 注文受付
- `shipped` — 発送済み
- `completed` — 完了
- `return_pending` — 返品処理中
- `returned` — 返品完了

**データ概要：** 30件のサンプル注文（2023年1月〜7月）

---

### raw_payments.csv（支払データ）

**場所：** `dbt_project/seeds/raw_payments.csv`

| 列名 | 意味 |
|------|------|
| id | 支払の一意なID |
| order_id | どの注文の支払か（raw_ordersのidと対応） |
| payment_method | 支払方法 |
| amount | 支払金額 |
| payment_date | 支払日 |
| status | 支払ステータス（success / fail） |

**支払方法の種類：** `credit_card`（クレジットカード）/ `bank_transfer`（銀行振込）/ `coupon`（クーポン）

**データ概要：** 35件の支払記録

---

## 2. ステージングモデル（models/staging/）

「ステージング」とは、生データを **分析しやすい形に整形する第1層** です。  
列名の統一、型変換、フラグ列の追加などを行います。ビュー（View）として作成されます。

> **ビューとは：** テーブルのようにSELECTできますが、実データは持たず、クエリのたびに元テーブルから取得します。ストレージを節約できます。

---

### stg_customers.sql（顧客ステージング）

**場所：** `dbt_project/models/staging/stg_customers.sql`  
**参照元：** `raw_customers`  
**出力先：** `analytics.stg_customers`（ビュー）

**主な加工内容：**
- `id` → `customer_id` に列名変更（意味をわかりやすく）
- `first_name + last_name` を連結して `full_name` を追加
- `created_at` → `customer_created_at` に列名変更

---

### stg_orders.sql（注文ステージング）

**場所：** `dbt_project/models/staging/stg_orders.sql`  
**参照元：** `raw_orders`  
**出力先：** `analytics.stg_orders`（ビュー）

**主な加工内容：**
- `id` → `order_id`、`status` → `order_status`、`amount` → `order_amount` に列名変更
- `is_completed` フラグを追加（status が `completed` なら true）

---

### stg_payments.sql（支払ステージング）

**場所：** `dbt_project/models/staging/stg_payments.sql`  
**参照元：** `raw_payments`  
**出力先：** `analytics.stg_payments`（ビュー）

**主な加工内容：**
- `id` → `payment_id`、`status` → `payment_status` などに列名変更
- `is_successful` フラグを追加（status が `success` なら true）

---

## 3. マートモデル（models/marts/）

「マート（Data Mart）」とは、**ビジネスロジックを組み込んだ分析用テーブル** です。  
ステージングモデルを結合・集計して、レポートやMetricFlowが使いやすい形に整えます。  
テーブル（Table）として作成されます（実データをDBに保存）。

---

### customers.sql（顧客集計テーブル）

**場所：** `dbt_project/models/marts/customers.sql`  
**参照元：** `stg_customers`、`stg_orders`  
**出力先：** `analytics.customers`（テーブル）

**どんなデータか：** 顧客ごとの注文サマリー

| 主な列 | 内容 |
|--------|------|
| customer_id | 顧客ID |
| full_name, email, country | 顧客基本情報 |
| total_orders | 総注文件数 |
| completed_orders | 完了済み注文件数 |
| lifetime_value | 累計注文金額（LTV） |
| first_order_date | 初回注文日 |
| most_recent_order_date | 最終注文日 |
| is_active | アクティブ顧客フラグ（1件でも注文があればtrue） |

---

### orders.sql（注文集計テーブル）

**場所：** `dbt_project/models/marts/orders.sql`  
**参照元：** `stg_orders`、`stg_customers`、`stg_payments`  
**出力先：** `analytics.orders`（テーブル）

**どんなデータか：** 注文ごとの詳細（顧客情報・支払情報を結合済み）

| 主な列 | 内容 |
|--------|------|
| order_id | 注文ID |
| customer_id, customer_name, country | 注文した顧客の情報 |
| order_date, order_status, order_amount | 注文情報 |
| is_completed | 完了フラグ |
| total_paid | 実際に支払われた金額合計 |
| payment_method | 主な支払方法 |

---

### metricflow_time_spine.sql（日付テーブル）

**場所：** `dbt_project/models/marts/metricflow_time_spine.sql`  
**出力先：** `analytics.metricflow_time_spine`（テーブル）

**どんなデータか：** 2010-01-01〜2030-12-31 の全日付を1行1日で持つテーブル

MetricFlowが「月別」「週別」など **時系列での集計** を行うために必要な特殊テーブルです。  
売上ゼロの日も含めて連続した日付一覧を保持することで、穴のない時系列グラフが作れます。

---

## 4. メトリクス定義（models/metrics/）

MetricFlowで使う指標定義をYAMLファイルで宣言します。  
SQLは一切書かずに、集計の意味だけを定義します。

---

### semantic_models.yml（セマンティックモデル）

**場所：** `dbt_project/models/metrics/semantic_models.yml`

セマンティックモデルとは「このテーブルのどの列が何を意味するか」を MetricFlow に教える定義です。

**ordersセマンティックモデル（orders テーブルを基に）**

| 定義種別 | 列名 | 意味 |
|---------|------|------|
| エンティティ | order_id | 主キー（1注文＝1行を識別） |
| エンティティ | customer_id | 外部キー（顧客との紐付け） |
| ディメンション | order_date | 時間軸（日/月/年で集計可能） |
| ディメンション | order_status | 分類軸（ステータス別に集計可能） |
| ディメンション | country | 分類軸（国別に集計可能） |
| ディメンション | payment_method | 分類軸（支払方法別） |
| メジャー | order_total | 注文金額の合計（SUM） |
| メジャー | order_count | 注文件数（COUNT DISTINCT） |
| メジャー | avg_order_value | 平均注文金額（AVERAGE） |
| メジャー | completed_order_count | 完了注文件数（条件付きSUM） |

**customersセマンティックモデル（customers テーブルを基に）**

| 定義種別 | 列名 | 意味 |
|---------|------|------|
| エンティティ | customer_id | 主キー |
| ディメンション | customer_created_at | 登録日（時間軸） |
| ディメンション | country | 国 |
| ディメンション | is_active | アクティブフラグ |
| メジャー | customer_count | 顧客数（COUNT DISTINCT） |
| メジャー | active_customer_count | アクティブ顧客数 |
| メジャー | total_lifetime_value | LTV合計（SUM） |
| メジャー | avg_lifetime_value | 平均LTV |

---

### metrics.yml（メトリクス定義）

**場所：** `dbt_project/models/metrics/metrics.yml`

定義された12種類のメトリクス：

| 種別 | メトリクス名 | 日本語ラベル | 説明 |
|------|------------|------------|------|
| Simple | revenue | 売上合計 | 注文金額の合計（order_totalを集計） |
| Simple | order_count | 注文件数 | 期間内の注文数 |
| Simple | average_order_value | 平均注文金額 | 1注文あたりの平均 |
| Simple | completed_orders | 完了注文件数 | 完了ステータスの注文数 |
| **Ratio** | completion_rate | 注文完了率 | 完了注文÷全注文 |
| Simple | total_customers | 顧客総数 | 登録顧客の総数 |
| Simple | active_customers | アクティブ顧客数 | 1件以上注文した顧客数 |
| **Ratio** | customer_activation_rate | 顧客活性化率 | アクティブ顧客÷全顧客 |
| Simple | average_customer_ltv | 平均顧客生涯価値 | 顧客1人あたりの平均LTV |
| **Cumulative** | cumulative_revenue | 累積売上 | 全期間の累積売上金額 |
| **Cumulative** | cumulative_customers | 累積顧客数 | 全期間の累積登録顧客数 |
| **Derived** | revenue_per_customer | 顧客1人あたり売上 | revenue ÷ active_customers |

**メトリクスの種別について：**
- **Simple（シンプル）** — セマンティックモデルのメジャーを直接集計
- **Ratio（比率）** — 2つのメトリクスの比（分子÷分母）
- **Cumulative（累積）** — 全期間の累積値を計算
- **Derived（派生）** — 複数メトリクスを数式で組み合わせたもの

## 5. スキーマファイル（schema.yml）

各モデルフォルダにある `schema.yml` は、モデルの **説明とテストを宣言するファイル** です。

**staging/schema.yml の例：**

```yaml
- name: stg_customers
  columns:
    - name: customer_id
      tests:
        - unique    # ← 重複なしを保証
        - not_null  # ← NULLなしを保証
    - name: order_status
      tests:
        - accepted_values:
            values: ['completed', 'returned', ...]  # ← 想定外の値を検出
```

このファイルがあることで `dbt test` コマンドがデータ品質チェックを実行できます。  
詳しくは [03_コマンド解説.md](03_コマンド解説.md) を参照してください。
